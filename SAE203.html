// 01 — Contexte

Présentation du projet

Cadre académique, objectif et contraintes techniques imposées.

Objectif
  • Développer une application web de gestion d'un drive de supermarché permettant aux clients de passer des commandes en ligne
  • Déployer l'application sur une infrastructure à deux VMs VirtualBox sous Debian
  • Travailler en équipe avec GitHub comme outil de collaboration
Contraintes imposées par le professeur
  • Tables SQL manuelles — Django ne crée pas les tables, elles sont créées manuellement en SQL (managed = False dans les Meta)
  • Stack imposée — Django (pas Flask), MariaDB/MySQL
  • Deux VMs obligatoires — une pour la BDD, une pour l'app Django
  • Repo GitHub unique pour tout le groupe
Répartition des rôles dans le groupe
  • Aymen — Schéma relationnel BDD, CRUD Produits + import CSV, CRUD Clients, CRUD Lignes de commande, calcul coût total
  • Armin — CRUD Commandes, modèles Commande/LigneCommande, interface HTML/CSS
  • Basile — Modèles Catégorie/Produit/Client, CRUD Catégories, configuration des deux VMs, déploiement MariaDB + Nginx/Gunicorn
Architecture de déploiement
  • VM1 — Debian + MariaDB (base : drive_db)
  • VM2 — Debian + Django + Nginx + Gunicorn (reverse proxy → socket Unix)
  • Communication inter-VMs via réseau VirtualBox accès par pont, IPs statiques sur le même sous-réseau
  • Fichiers statiques servis par Nginx via collectstatic

Extrait de code — méthodes de calcul
# models.py — calcul du coût sans stockage BDD class LigneCommande(models.Model): commande = models.ForeignKey(Commande, on_delete=models.CASCADE) produit = models.ForeignKey(Produit, on_delete=models.PROTECT) quantite = models.IntegerField() def sous_total(self): return self.quantite * self.produit.prix class Meta: db_table = 'ligne_commande' # table créée manuellement en SQL managed = False class Commande(models.Model): def cout_total(self): return sum(l.sous_total() for l in self.lignecommande_set.all())

// 04 — Difficultés & solutions

Problèmes rencontrés

Démarche de résolution des problèmes techniques concrets rencontrés durant le projet.

Conflit Git
Conflit de migrations lors de la fusion avec le code d'Armin

Armin avait nommé son modèle LigneProduit alors que le projet utilisait LigneCommande. Les fichiers de migration étaient incompatibles. Solution : suppression complète des fichiers de migration (0001, 0002) et du fichier db.sqlite3, puis makemigrations + migrate à zéro.

Déploiement
mysqlclient refusait de s'installer sur la VM Debian

pip install mysqlclient échouait avec une erreur de compilation C. Cause : bibliothèques système manquantes. Solution : apt install pkg-config libmariadb-dev avant l'installation pip.

Django
STATIC_ROOT non défini — collectstatic échoue

La commande python manage.py collectstatic refusait de s'exécuter. Cause : STATIC_ROOT absent de settings.py. Solution : ajout de STATIC_ROOT = BASE_DIR / 'staticfiles'.

Réseau VMs
VM1 et VM2 ne communiquaient pas — IPs sur des sous-réseaux différents

La VM MariaDB avait une IP statique configurée sur un sous-réseau différent de la VM Django. La connexion Django → MariaDB échouait. Solution : reconfiguration de /etc/network/interfaces sur les deux VMs pour placer les deux IPs sur le même sous-réseau (192.168.x.0/24).

// 07 — English summary

Project overview

English summary

SAÉ 2.03 was a team project focused on building a full-stack web application for a supermarket drive (online grocery ordering system). Working in a group of three, we designed and deployed a Django application connected to a MariaDB database, hosted across two Debian virtual machines running on VirtualBox.

My personal contributions covered the database relational schema design (5 tables modelled on draw.io), full CRUD operations for Products — including a CSV bulk-import feature using Python's csv.DictReader with error handling — Customers CRUD, and Order Lines CRUD with on-the-fly cost calculation methods (sous_total() and cout_total()) computed without storing redundant data in the database.

A key technical constraint imposed by the professor was that Django must not create database tables — all tables were created manually using raw SQL, with Django models set to managed = False. The application was deployed using Nginx as a reverse proxy and Gunicorn as a WSGI server communicating through a Unix socket. I also participated in troubleshooting deployment issues: resolving a missing system library blocking mysqlclient installation, fixing a STATIC_ROOT misconfiguration, and diagnosing a network routing issue preventing communication between the two VMs on different subnets.

This project strengthened my skills in web application development, relational database design, collaborative version control with Git, and Linux server deployment — all core competencies of the BUT Réseaux & Télécommunications program.
